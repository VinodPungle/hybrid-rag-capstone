## 1. What This Solution Does

This solution is a **Hybrid RAG (Retrieval-Augmented Generation)** system that lets users upload PDF documents and ask natural-language questions about them. It combines two search methods for better accuracy:

- **Vector Search** — finds semantically similar text chunks using FAISS embeddings
- **Knowledge Graph Search (GraphRAG)** — traverses entity relationships extracted into a Neo4j graph database

The answers are generated by Azure OpenAI's GPT model using context from both search methods, reducing hallucination and improving relevance.

**End Use Cases:**
- Document understanding and Q&A (audit reports, compliance guides, research papers)
- Knowledge extraction from large PDF documents
- Enterprise document intelligence with traceable source context
- Demonstration of LLMOps practices for production-grade LLM applications

## 3. Architecture Components

### 3.1 Document Ingestion

**What it does:** Takes a PDF file, extracts raw text, and splits it into overlapping chunks suitable for embedding and retrieval.

**Why it matters:** LLMs have limited context windows. Chunking breaks large documents into manageable pieces while overlap ensures important information at chunk boundaries isn't lost.

**Implementation:**

| File | Purpose |
|------|---------|
| `ingestion/pdf_loader.py` | Extracts text from PDF using `pypdf` library |
| `ingestion/chunker.py` | Splits text into word-based chunks with configurable size and overlap |

**Key parameters** (from `config/config.yaml`):
```yaml
ingestion:
  chunk_size: 500      # words per chunk
  chunk_overlap: 80    # overlapping words between consecutive chunks
```

**How chunking works:**
```
Document text: "word1 word2 word3 word4 word5 word6 word7 word8 word9 word10 ..."

chunk_size=5, overlap=2:
  Chunk 1: "word1 word2 word3 word4 word5"
  Chunk 2: "word4 word5 word6 word7 word8"    ← 2-word overlap with Chunk 1
  Chunk 3: "word7 word8 word9 word10 ..."     ← 2-word overlap with Chunk 2
```

---

### 3.2 Embedding Generation

**What it does:** Converts text chunks into high-dimensional numerical vectors (embeddings) that capture semantic meaning. Similar texts produce similar vectors.

**Why it matters:** Embeddings enable semantic search — finding relevant chunks based on meaning rather than keyword matching.

**Implementation:**

| File | Purpose |
|------|---------|
| `embeddings/embedder.py` | Calls Azure OpenAI's embedding API to generate 1536-dimensional vectors |

**How it works:**
```
"The audit committee oversees financial reporting"
        │
        ▼  Azure OpenAI Embedding API
[0.0123, -0.0456, 0.0789, ..., 0.0234]   ← 1536-dimensional vector
```

**Configuration:**
```yaml
embedding:
  api_version: "2024-02-01"
  dimensions: 1536
```

**Environment variables required** (in `.env`):
```
AZURE_OPENAI_ENDPOINT=<your-azure-endpoint>
AZURE_OPENAI_API_KEY=<your-api-key>
AZURE_EMBEDDING_DEPLOYMENT=<embedding-model-name>
```

---

### 3.3 Vector Store (FAISS)

**What it does:** Stores embedding vectors in a FAISS (Facebook AI Similarity Search) index for fast nearest-neighbor lookup.

**Why it matters:** When a user asks a question, their query is embedded into the same vector space and FAISS finds the most similar document chunks in milliseconds.

**Implementation:**

| File | Purpose |
|------|---------|
| `vector_db/faiss_store.py` | Builds and queries a FAISS `IndexFlatL2` index |
| `retrieval/search.py` | Performs vector similarity search against the FAISS index |

**How vector search works:**
```
User query: "What does the audit committee do?"
        │
        ▼  Embed query
[0.0111, -0.0333, 0.0555, ...]   ← query vector
        │
        ▼  FAISS nearest-neighbor search (L2 distance)
Top-3 most similar chunks returned
```

**Configuration:**
```yaml
retrieval:
  vector_top_k: 3   # number of results to return
```

---

### 3.4 Knowledge Graph (Neo4j — GraphRAG)

**What it does:** Extracts entities (people, organizations, committees, etc.) and their relationships from document chunks using an LLM, then stores them in a Neo4j graph database. This enables relationship-aware search that vector search alone cannot provide.

**Why it matters:** Vector search finds text that is semantically similar to the query, but misses structured relationships. For example, "Who reports to the audit committee?" requires understanding organizational relationships, which is what the knowledge graph provides.

**Implementation:**

| File | Purpose |
|------|---------|
| `graph_db/entity_extractor.py` | Uses Azure OpenAI to extract entities and relationships from each chunk as structured JSON |
| `graph_db/neo4j_store.py` | Stores entities as nodes and relationships as edges in Neo4j; provides query functions |
| `retrieval/graph_search.py` | Searches the knowledge graph for entities matching the user's query |

**Entity extraction flow:**
```
Chunk text: "The Audit Committee reports to the Board of Directors and
             oversees the Internal Audit function."
        │
        ▼  LLM with extraction prompt
{
  "entities": [
    {"name": "Audit Committee", "type": "Committee"},
    {"name": "Board of Directors", "type": "Organization"},
    {"name": "Internal Audit", "type": "Process"}
  ],
  "relationships": [
    {"source": "Audit Committee", "target": "Board of Directors", "relation": "reports to"},
    {"source": "Audit Committee", "target": "Internal Audit", "relation": "oversees"}
  ]
}
        │
        ▼  Neo4j
(Audit Committee)──[reports to]──>(Board of Directors)
(Audit Committee)──[oversees]──>(Internal Audit)
```

**Configuration:**
```yaml
entity_extraction:
  temperature: 0.0              # deterministic extraction
  max_tokens: 1000
  entity_types:                  # types the LLM should look for
    - Person
    - Organization
    - Committee
    - Role
    - Regulation
    - Process
    - Document
    - Concept
```

**Environment variables required** (in `.env`):
```
NEO4J_URI=neo4j+s://<your-neo4j-instance>
NEO4J_USER=neo4j
NEO4J_PASSWORD=<your-password>
AZURE_OPENAI_DEPLOYMENT=<gpt-model-name>
```

### 3.5 Hybrid Search (Vector + Graph)

**What it does:** Combines results from both vector search and knowledge graph search into a unified context for answer generation.

**Why it matters:** Neither search method alone is sufficient:
- Vector search is great for semantic similarity but misses structured relationships
- Graph search captures relationships but may miss relevant text passages
- **Hybrid search combines both** for more complete and accurate retrieval

**Implementation:**

| File | Purpose |
|------|---------|
| `retrieval/hybrid_search.py` | Orchestrates both searches and merges results into `combined_context` |

**How hybrid search works:**
```
User query: "What are audit committee responsibilities?"
        │
        ├──> Vector Search (FAISS)
        │    Returns: top-3 similar text chunks
        │
        └──> Graph Search (Neo4j)
             Returns: entity relationships matching query keywords
        │
        ▼  Merge
combined_context = vector_chunks + "\n" + graph_relationships
```

**Graceful degradation:** If Neo4j is unavailable, the system falls back to vector-only search without crashing.

### 3.6 LLM Answer Generation

**What it does:** Sends the user's question along with the retrieved context to Azure OpenAI's GPT model and returns a grounded answer.

**Why it matters:** The LLM generates human-readable answers based only on the retrieved context, not its training data. This is the "Generation" in RAG — it reduces hallucination by constraining the model to answer from provided evidence.

**Implementation:**

| File | Purpose |
|------|---------|
| `llm/generator.py` | Calls Azure OpenAI's chat completion API with the prompt template from config |

**Prompt structure** (from `config.yaml`):
```
System: You are a helpful document understanding assistant.
        Answer questions based only on the provided context.

User:   Context: [retrieved chunks + graph relationships]
        Question: [user's query]
        Answer:
```

**LLMOps tracking:** Every LLM call records:
- Token usage (prompt + completion tokens) via structured logging
- Latency (wall-clock time) via structured logging
- Prometheus metrics for monitoring dashboards

### 3.7 YAML Configuration Management

**What it does:** Centralizes all tunable, non-secret parameters in a single `config/config.yaml` file, loaded by a singleton config loader.

**Why it matters (LLMOps):** In production LLM systems, you frequently need to adjust parameters like chunk size, temperature, prompt templates, and search limits without changing code. A YAML config file enables this.

**Implementation:**

| File | Purpose |
|------|---------|
| `config/config.yaml` | All non-secret configuration parameters |
| `config/settings.py` | Singleton loader with `get(section, key)` API |

**Usage pattern throughout the codebase:**
```python
from config.settings import get as cfg

# Get an entire section
llm_config = cfg("llm")

# Get a specific key
top_k = cfg("retrieval", "vector_top_k")   # returns 3
```

**Design principle:** Secrets (API keys, passwords) stay in `.env`; everything else goes in `config.yaml`.

### 3.8 Structured Logging

**What it does:** Provides consistent, parseable log output across the entire application using Python's logging module with a structured format.

**Why it matters (LLMOps):** In production, you need machine-parseable logs for debugging, auditing, and alerting. Structured logs replace scattered `print()` statements with consistent, leveled output that can be sent to log aggregation tools.

**Implementation:**

| File | Purpose |
|------|---------|
| `utils/logger.py` | `get_logger(name)` factory, `log_llm_call()` for LLM tracking, `timed()` decorator |

**Log format:**
```
2025-03-15 10:23:45 | INFO | llm.generator | Query answered in 2.34s
2025-03-15 10:23:45 | INFO | llm.generator | LLM call_type=generation prompt_tokens=850 completion_tokens=120 latency=2.34s
```

**Configurable tracking flags** (from `config.yaml`):
```yaml
llmops:
  monitoring:
    log_prompts: false        # log full prompts/responses (enable for debugging)
    track_token_usage: true   # log token counts per LLM call
    track_latency: true       # log wall-clock time per LLM call
```

### 3.9 REST API (FastAPI)

**What it does:** Exposes the RAG pipeline as a REST API with three endpoints, making it accessible to any client (web apps, mobile apps, scripts).

**Why it matters:** A REST API decouples the pipeline from any specific UI and enables integration with other systems.

**Implementation:**

| File | Purpose |
|------|---------|
| `api/app.py` | FastAPI application with endpoints, Pydantic models, and Prometheus instrumentation |

**Endpoints:**

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/health` | Health check — is the service running and is a document loaded? |
| `POST` | `/ingest` | Upload a PDF → builds vector index + knowledge graph |
| `POST` | `/ask` | Ask a question → returns answer with source context |
| `GET` | `/metrics` | Prometheus metrics endpoint (auto-generated) |

**Request/Response examples:**

```bash
# Upload a document
curl -X POST http://localhost:8000/ingest -F "file=@document.pdf"

# Response:
{
  "chunks": 42,
  "entities": 15,
  "relationships": 23,
  "message": "Document ingested successfully in 12.3s."
}

# Ask a question
curl -X POST http://localhost:8000/ask \
  -H "Content-Type: application/json" \
  -d '{"query": "What are audit committee responsibilities?"}'

# Response:
{
  "answer": "The audit committee is responsible for...",
  "vector_results": ["chunk1...", "chunk2...", "chunk3..."],
  "graph_results": ["Audit Committee --[oversees]--> Financial Reporting"],
  "latency_s": 3.456
}
```

**Swagger UI:** Available at `http://localhost:8000/docs` for interactive API testing.

### 3.11 Evaluation Metrics (BLEU / ROUGE / Faithfulness)

**What it does:** Measures the quality of RAG-generated answers against reference (gold-standard) answers using standard NLP metrics.

**Why it matters (LLMOps):** You need quantitative metrics to track answer quality over time, compare prompt versions, and detect regressions when changing pipeline parameters.

**Implementation:**

| File | Purpose |
|------|---------|
| `evaluation/metrics.py` | `bleu_score()`, `rouge_scores()`, `faithfulness_score()`, `evaluate()` |
| `evaluation/run_eval.py` | Runner script that builds the pipeline and evaluates test cases |
| `evaluation/test_cases.json` | Test question-answer pairs for evaluation |

**Metrics explained:**

| Metric | What it Measures | Range |
|--------|-----------------|-------|
| **BLEU-1 to BLEU-4** | N-gram overlap between generated and reference answers. BLEU-4 (4-gram) is strictest. | 0.0 – 1.0 |
| **ROUGE-1, ROUGE-2, ROUGE-L** | Recall-oriented overlap. ROUGE-L uses longest common subsequence. Returns precision, recall, and F1. | 0.0 – 1.0 |
| **Faithfulness** | Fraction of generated sentences that are grounded in the source context (not hallucinated). | 0.0 – 1.0 |

**Which metrics run is controlled by config:**
```yaml
llmops:
  evaluation:
    enabled: true
    metrics:
      - bleu
      - rouge
      - faithfulness
```

### 3.13 Pipeline Orchestration (Airflow)

**What it does:** Defines the RAG pipeline as an Airflow DAG (Directed Acyclic Graph) with 7 tasks that can be triggered manually or on a schedule.

**Why it matters (LLMOps):** Production pipelines need orchestration for scheduling, retry logic, dependency management, and monitoring. Airflow provides all of this with a web UI.

**Implementation:**

| File | Purpose |
|------|---------|
| `airflow/dags/rag_pipeline_dag.py` | 7-task DAG with parallel branches |

**DAG structure:**
```
load_document → chunk_text → ┬─ create_embeddings → build_index    ──┬─→ run_evaluation
                              └─ extract_entities  → build_graph    ──┘
```

After chunking, the pipeline splits into two parallel branches:
- **Branch 1 (Vector path):** embed chunks → build FAISS index
- **Branch 2 (Graph path):** extract entities → build Neo4j graph

Both branches must complete before evaluation runs.

## 5. Data Flow — End to End

Here is the complete data flow from document upload to answer generation:

```
Step 1: INGESTION
  PDF file
    → pdf_loader.py extracts raw text
    → chunker.py splits into overlapping chunks (500 words, 80 overlap)

Step 2: EMBEDDING (Branch 1)
  Chunks
    → embedder.py sends chunks to Azure OpenAI embedding API
    → Returns 1536-dimensional vectors per chunk

Step 3: INDEXING (Branch 1)
  Vectors
    → faiss_store.py builds an IndexFlatL2 FAISS index
    → Enables millisecond nearest-neighbor search

Step 4: ENTITY EXTRACTION (Branch 2)
  Chunks
    → entity_extractor.py sends chunks to Azure OpenAI GPT with extraction prompt
    → Returns structured JSON: entities + relationships

Step 5: GRAPH BUILD (Branch 2)
  Entities + Relationships
    → neo4j_store.py creates nodes and edges in Neo4j
    → Enables relationship traversal queries

Step 6: QUERY
  User asks a question
    → hybrid_search.py runs both:
        a) Vector search: embed query → FAISS top-k → return similar chunks
        b) Graph search: extract keywords → match entities in Neo4j → return relationships
    → Merges into combined_context

Step 7: GENERATION
  Combined context + question
    → generator.py sends to Azure OpenAI GPT with prompt template
    → Returns grounded answer

Step 8: EVALUATION (optional)
  Generated answer vs reference answer
    → metrics.py computes BLEU, ROUGE, faithfulness scores
```

## 11. Troubleshooting

### Common issues and fixes:

**"INPUT_FILE environment variable is not set"**
- Create a `.env` file with `INPUT_FILE=data/raw/your-document.pdf`
- Make sure the file exists at that path

**"Form data requires python-multipart"**
- Run: `pip install python-multipart`

**Neo4j connection errors**
- Check `NEO4J_URI`, `NEO4J_USER`, `NEO4J_PASSWORD` in `.env`
- If using Neo4j Aura, the `ssl_fix: true` in config.yaml handles certificate issues
- The system gracefully falls back to vector-only search if Neo4j is unavailable

**Azure OpenAI errors**
- Verify `AZURE_OPENAI_ENDPOINT`, `AZURE_OPENAI_API_KEY` in `.env`
- Check that deployment names match: `AZURE_OPENAI_DEPLOYMENT` (chat model), `AZURE_EMBEDDING_DEPLOYMENT` (embedding model)
- Ensure your Azure subscription has quota for the deployed models

**Tests failing with import errors**
- Make sure you're running from the project root: `python -m pytest tests/ -v`
- Check that `conftest.py` exists at the project root
- Verify `pytest.ini` has `pythonpath = .`

**Docker Compose errors**
- Ensure Docker Desktop is running
- `.env` file must exist (Docker Compose reads it via `env_file`)
- Use `docker compose logs <service>` to check individual service logs
